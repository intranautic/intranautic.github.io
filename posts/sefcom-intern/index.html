SEFCOM Research Internship

Jul 9, 2023

This summer I had decided to attend a high school research internship at the SEFCOM lab with Zion Basque also known as mahaloz serving as my mentor. Within this post I will document my work and findings throughout the course of the internship, as well as various insights I have acquired as a result.

My work throughout the course of the internship has largely centered around implementing additional optimization and simplification passes in the angr decompiler with the goal of correctness and improving control flow structuring decisions in decompilation.

Background

As a bit of background, decompilation1 is the process of translating and deoptimizing low level code into higher level constructs. This allows a human to more efficiently analyze such abstractions which serves to aid in the process of static binary analysis and source code recovery.

One of the primary hardships in the problem of decompilation is the fact that compilation itself is a directed lossy process. Higher level and abstract code constructs dont exist at a lower level. In the process of compilation, information is invariably lost as a result such as type information, comments, preprocessor directives (inlined and expanded before compilation), symbol names (when stripped) and more.

Aggressive optimization can also potentially mutate the original structure and control flow of the program into an entirely separate CFG, yet still manage to preserve the same semantic properties as the original. This can lead to less accurate decompilation output due to the fact that structuring relies heavily on the accuracy of Control Flow Recovery.

The decompilation process generally follows a similar pipeline documented below.

  1. Lifting Phase
  • The first step of the decompilation phase involves the disassembly and conversion of low level code into an intermediate representation (IR) which generally preserves the semantics of the original program.
  • As a result of this lifting phase, the decompilation engine can be portable by operating on our intermediate representation as opposed to each and every single architecture we decide we want to support.
  1. Control Flow Analysis2
  • This next process within the decompilation pipeline concerns itself with control flow graph recovery. The analysis generally concerns itself with constructing basic blocks from the lifted intermediate representation and attaching directed edges to represent control flow.
  • Within the recovered control flow graph, basic blocks represent vertices and edges represent control flow within the program.
  • This step within the decompilation process is crucial as any discrepancies can lead to drastic alterations within the resulting output. There are various algorithms for control flow graph recovery which are known to work well in practice.
  1. Variable and Type Analysis
  • The next step involves itself with recovering vital information of the variables within our program. Within a relatively high level language, types represent sets of values which serve to impose semantic constraints on symbolic values or expressions. However at a low level, the microprocessor is only concerned with the size and sign of a variable.
  • Variable Analysis and Recovery is generally done by assigning symbols to regions of memory which may potentially represent variables within the program. Such variables can include memory addresses, registers and et cetera.
  • The process of Type Analysis and Recovery is generally done via a technique known as Value Set Analysis (VSA). This process utilizes the various operations and statements performed on recovered variable to impose semantic constraints. This process is fully documented within the paper, TIE: Principled Reverse Engineering of Types in Binary Programs3.
  • Something to keep in mind is the fact that a symbolic or value can potentially represent different types at different locations within the CFG. The process of value set analysis is to deduce the set of all potential types which it can represent.
  1. Control Flow Structuring45
  • As previously stated, high level control flow constructs and abstractions are lost at compile time. This phase within decompilation concerns itself with the recovery of constructs such as conditional statements, switch cases and loops.
  • The structuring decisions made in this phase drastically affect the resulting output.
  • There are various structuring algorithms which are documented within the papers provided above. The problem of structuring is still an ongoing issue as compiler optimization can cause this phase within decompilation to fail miserably.
  • This is just one example of how structuring decisions can ultimately affect the accuracy of the resulting decompilation. One simple and well known algorithm for Control Flow Structuring is to match specific regions of the CFG to predefined control flow schemas. This is the reason as to why accurate Control Flow Recovery is so impactful, as the structuring phase is highly dependent on pattern matching.
  1. High-Level Structured Code Generation
  • The last stage in the decompilation pipeline concerns itself with actually converting our recovered intermediate representation into C style source code (or whichever high level language we are targeting).
  • Additional optimizations and simplification passes are also performed at this stage to improve the recovered source code.

Structured C Code Formatting

My first tasks generally revolved improving the structured C code generation by implementing simplification passes which served to refactor the output to adopt a more C-style format. The features I had implemented are cstyle_ifs and cstyle_nullcmp, both of which serve to reformat the resulting structured C code.

Each of these reformatting don’t actually affect structuring decisions, below is an example of both features.

int main(void) {
  if (x == 0) {
    return 1;
  }
  return 0;
}

This code is fine, but we can also represent it in a semantically equivalent form as follows which follows a more C-style format.

int main(void) {
  if (!x)
    return 1;
  return 0;
}

We can see that it converts the condition to utilize the unary negation operator and removes the braces in the presence of a single statement. Each are fairly simple aesthetic adjustments, but it makes the resulting decompilation look much more readable in my opinion.

Control Flow Structuring

The next feature I had implemented concerns itself with implementing an optimization to improving structuring decisions which serve to improve the accuracy and correctness, but before we proceed I would first like to establish some prerequisites.

There are a few semantically sound ways of structuring a CFG, each with varying tradeoffs and accuracy. The problem of Control Flow Structuring in this case concerns itself with what decisions to make and when.

A crucial reason as to why our decompiler may potentially inaccurately perform structuring is due to the accuracy of the CFG. Say for example, during the process of performing Control Flow Analysis and building the graph: the decompiler does not properly interpret a basic block or statement which either returns from the subroutine or exits.

This will potentially affect control flow, as if say function exit(0) had not been interpreted as a noreturn, it would be semantically correct to wrap the false branch in an else scope. However this isn’t the case in the example provided, which is what this optimization pass intends to mend.

CFG #1 CFG #2

The results don’t look particularly drastic in the examples provided but at a large enough scale, they can drastically reduce the accuracy of the decompilation and make program analysis quantitatively more difficult.

To effectively implement this optimization into the angr decompiler, we have to consider and handle another case shown below.

One problem we encountered is the fact that the compiler could very well just flip a comparison into a semantically equivalent alternative in an optimization pass. It could also be due to the underlying implementation of the binary analysis framework angr and how it lifts comparisons into a higher level intermediate representation.

Whichever causes the issue, the core problem remains that it affects the accuracy of our decompilation. The following is an example which demonstrates how the sign of a comparison could potentially affect Control Flow Analysis and the resulting CFG.

Original Source CFG Decompilation CFG

As we can see, the resulting structure of the decompilation does not resemble the original source program. They are semantically equivalent, however in this scenario, we can consider the structuring decision made by the decompiler to be incorrect.

This is what my feature attempts to mediate and in doing so allows the resulting structured C code generation to resemble the original source. So how is this optimization implemented and how does it work?

Lets use the following example as a means of demonstrating the process in which the optimization pass takes.

Original decompilation output

int main(int argc, char** argv) {
  if (argc)
    putchar('a');
  else
    return 1;
  return 0;
}

The optimization flips the true and false branches of the condition node and negates the condition within the comparison iff the else branch does not contain any successors.

int main(int argc, char** argv) {
  if (!argc)
    return 1;
  else
    putchar('a');
  return 0;
}

Now due to the fact that the true branch does not return, it will never fall through so the else statement is redundant. Due to this property we can completely remove the else branch.

int main(int argc, char** argv) {
  if (!argc)
    return 1;
  putchar('a');
  return 0;
}

Metrics

The example provided is very minute, however when analyzing large binaries, such a feature scales by reducing complexity of the resulting decompilation and in turn saving the reverse engineer time and cognitive load.

We can utilize two relevant metrics for evaluating impact of this feature.

  1. The total sum of instances in which the feature triggers.
  2. The accuracy of the decompilation to the original source.

We can measure the first quantitative metric by instrumenting the feature within angr and running against the coreutils test suite, this feature was triggered a total of 496 times.

For our second metric, we can measure accuracy of our decompilation by comparing between angr and Binary Ninja’s decompilers with the original source to measure correctness and accuracy.
